One of the nice things about Go which I like it very much is that Go has lots of useful built-in packages that help us build software without relying heavily on external packages. Talking about building software, database operations consumes large of our time during development. There are several popular packages can help us reducing development time when working with database like gorm or Facebook ent, but still, working with only built-in database package is enough for most of the cases.

Go has built-in package database/sql which provide a generic interface around SQL database.

Establish a database connection

To start, let's create a sql.DB instance which represents a pool of connections. database/sql package does not include any drivers so in order to use a specific driver, we have to import it ourself. In the example bellow, I use mysql driver from go-sql-driver, when this module is import, it registers itself to database/sql's driver list.

 1// https://golang.org/src/database/sql/sql.go
 2
 3// Register makes a database driver available by the provided name.
 4// If Register is called twice with the same name or if driver is nil,
 5// it panics.
 6func Register(name string, driver driver.Driver) {
 7	driversMu.Lock()
 8	defer driversMu.Unlock()
 9	if driver == nil {
10		panic("sql: Register driver is nil")
11	}
12	if _, dup := drivers[name]; dup {
13		panic("sql: Register called twice for driver " + name)
14	}
15	drivers[name] = driver
16}
17
18// https://github.com/go-sql-driver/mysql/blob/master/driver.go
19func init() {
20	sql.Register("mysql", &MySQLDriver{})
21}

The list of the database drivers can be found here.

 4	"database/sql"

Calling Open may not actually create a connection, we can use Ping to establish a connection to verify if the database connection is alive. For most of the exported methods in database/sql package, there are two versions: one without context and the other has context in its parameters. If we use the first one, context.Background() will be used under the hood.

Working with database

Everytime we call Query or Exec method on *DB, it asks the connection poll for a connection which is active but idle or ask the Driver to create a new connection if there are no connections available in the pool.

Query data

Data from database can be queried by using combination of db.Query and rows.Scan function.

 1db, err := sql.Open("mysql", "root:@tcp(127.0.0.1:3306)/experiment?parseTime=true")
 2if err != nil {
 3    log.Fatal(err)
 4}
 5
 6log.Print("Get all users data:")
 7rows, err := db.Query("select * from users")
 8if err != nil {
 9    log.Fatal(err)
10}
11defer rows.Close()
12
13for rows.Next() {
14    var u User
15    err := rows.Scan(&u.id, &u.name, &u.email, &u.createdAt)
16    if err != nil {
17        log.Fatal(err)
18    }
19    log.Println(u)
20}

Execute SQL query

Sometimes, we need to execute some SQLs that do not return rows value like INSERT or DELETE. In this case, we better use Exec method, which returns a Result struct.

 1// https://golang.org/src/database/sql/sql.go
 2// A Result summarizes an executed SQL command.
 3type Result interface {
 4	// LastInsertId returns the integer generated by the database
 5	// in response to a command. Typically this will be from an
 6	// "auto increment" column when inserting a new row. Not all
 7	// databases support this feature, and the syntax of such
 8	// statements varies.
 9	LastInsertId() (int64, error)
10
11	// RowsAffected returns the number of rows affected by an
12	// update, insert, or delete. Not every database or database
13	// driver may support this.
14	RowsAffected() (int64, error)
15}

Usage example

 2if err != nil {

Transactions

Go package database/sql has struct Tx which represent a sql transaction, we can start a transaction by calling BeginTx method or just Begin if you decide to use context.Background() and passing no options. There are only two options for starting a transaction: Isolation level and ReadOnly.

1// https://golang.org/src/database/sql/sql.go
2
3// TxOptions holds the transaction options to be used in DB.BeginTx.
4type TxOptions struct {
5    // Isolation is the transaction isolation level.
6    // If zero, the driver or database's default level is used.
7    Isolation IsolationLevel
8    ReadOnly  bool
9}

To ensure data integrity, I recommend to use LevelReadCommited for isolation level for most of the cases and seeking for appropriate isolation level when there are some special requirements.

 1db, err := sql.Open("mysql", "root:@tcp(127.0.0.1:3306)/experiment?parseTime=true")
 2if err != nil {
 3    log.Fatal(err)
 4}
 5db.SetConnMaxLifetime(time.Minute * 3)
 6db.SetMaxOpenConns(10)
 7db.SetMaxIdleConns(10)
 8
 9txOpt := sql.TxOptions{
10    Isolation: sql.LevelReadCommitted,
11    ReadOnly:  false,
12}
13tx, err := db.BeginTx(context.Background(), &txOpt)
14if err != nil {
15    log.Fatal(err)
16}
17defer tx.Commit()
18
19rows, err := tx.Query("select * from users")
20if err != nil {
21    tx.Commit()
22    log.Println("Commit the transaction since there is nothing to do here.")
23    log.Fatal(err)
24}
25
26// Collect all users into list and update their name to be uppercase
27// Rollback if any of the update fails
28var users []user
29for rows.Next() {
30    var u user
31    err := rows.Scan(&u.id, &u.name, &u.email, &u.createdAt)
32    if err != nil {
33        tx.Rollback()
34        log.Fatalf("Rollback due to error: %v", err)
35    }
36    users = append(users, u)
37}
38rows.Close()
39for _, u := range users {
40    _, err = tx.Exec("update users set name = ? where id = ?", strings.ToUpper(u.name), u.id)
41    if err != nil {
42        tx.Rollback()
43        log.Fatalf("Rollback due to error: %v", err)
44    }
45}

One thing to keep in mind is that using transaction can leads to driver: bad connection easily if we do not use it properly. In the example code above, I have to collect all users from query result, then close the rows before start any updating operation. A transaction holds only a connection so because rows owns the current transaction's connection, we need to release rows's connection first, return it back to transaction, before doing any other operations.

Miscs

Refactor scanning rows

Read database result from query rows can be repetitive, we can move this logic into dedicate functions with rows.Scan as a parameter.

1func getUserData(scan func(dest ...interface{}) error) (*user, error) {
2    var u user
3    err := scan(&u.id, &u.name, &u.email, &u.created_at)
4    if err != nil {
5        return nil, err
6    }
7    return &u, nil
8}

We can pass rows.Scan function to getUserData to get user struct instance back

1for rows.Next() {
2    u, err := getUserData(rows.Scan)
3    if err != nil {
4            log.Fatal(err)
5    }
6    log.Println(u)
7}

Migration

Database migration can be done by using go-migrate tool. Migration file can be created by using following command.

1migrate create -ext sql -dir FOLDER_MIGRATION_FILES -seq TITLE

Then run the migration

1migrate -database YOUR_DATABASE_URL -path PATH_TO_YOUR_MIGRATIONS up

Check the Getting Started guide for more details.

Ending

Life will be much easier.
