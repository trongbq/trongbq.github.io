Go testing methods

23/10/2020

There is a section in software development which is often overlooked, developers find it boring and try to avoid it. That’s right, I am talking about writing tests. Let’s stop for a moment to see why testing is important and should be considered as an important step and deserved more time to write and think about it. What are the good things that writing test brings to us?

  • Confidence: Tests give us the confidence that our code run as we expected, it also help during refactoring. We will have much confidence if the code which is being modified is covered by test cases, so there will be no incorrect behaviors after refactoring.
  • Clean code: A lot of smell code can be detected when tests are being added, other developers ask for my help on how to write test when they found some trouble with finding the way to test their code. The problem always come from the way they write their code. The code is clean when writing test is easy and feasible without much tweaking around to make it testable.

Go has testing package which is super useful to help us writing tests in our Go program.

Basic testing

testing package provides support for automated testing, this package is used along with command go test to run execute any test functions. There are two common methods to signal failure of test cases are Error and Fatal. The different between Error and Fatal is that Error just marks the test is failed while Fatal exit the goroutine rightaway.

A group of test functions should be placed in a file that has name ends with _test.go. Generally, for every Go files in a repository, there is a test file for it in the same directory.

Note that package of test files can be same as targeted package or ends with <package>_test. If we follow the first option, test function can access both exported and non-exported identifiers.

A simple test function follow format like this:

func TestAdd(t *testing.T) {
    got := Add(3, 1)
    if got != 4 {
        t.Errorf("Failed Add(3, 1); want 4, got %v", got)
    }
}

How to run the test

To run test functions we use go test command, there are two different modes.

  1. Local directory mode: in this mode, go test is involked without package argument, it compiles and test all packages in current directory. Cache is disabled in this mode.
  2. Package list mode: go test is invoked with package argument. In this mode, go test still compiles and test all matched packages with cache enabled. Package argument can be an explicit package name like go test database/sql or go test . to test packages of current directory and go test ./... to test all packages in current directory tree.

Beside two modes above, we can also run individual test case by passing its name into go test like this go test -run <test function name>.

Test coverage

Go test provides excellent coverage generation when running tests. go test -cover generates simple coverage statistic after all packages are tested.

$ go test -cover database/sql
ok  	database/sql	1.300s	coverage: 84.4% of statements

Beside a simple coverage generation, it can also generate a full report on test coverage and we can use go tool to serve this file and view it on browser.

$ go test -coverprofile=prof.out
$ go tool cover -html-prof.out

Table-driven testing

To verify if a function behaves correctly, we need to feed it with different values of input and expect to receive corresponding output. Go has table-driven test which can be used to set up all the inputs and expected outputs to a single test, a lot of duplication can be reduced using this method. There are two things need to do when writing this kind of test. Firstly, we need to define an array of testing data, each item in that array will be used for a single test run. Secondly, using testing#Run method to define subtests within a single test to create a hierachical tests.

package add

import (
	"fmt"
	"testing"
)

func TestAdd(t *testing.T) {
	data := []struct {
		a    int
		b    int
		want int
	}{
		{
			a:    1,
			b:    1,
			want: 2,
		},
		{
			a:    -5,
			b:    3,
			want: -2,
		},
		{
			a:    9999,
			b:    -9999,
			want: 0,
		},
	}
	for _, test := range data {
		t.Run(fmt.Sprintf("Add(%v, %v)", test.a, test.b), func(tt *testing.T) {
			got := Add(test.a, test.b)
			if got != test.want {
				tt.Errorf("Fail Add(3, 1), want %v, got %v", test.want, got)
			}
		})
	}
}

When executing a table-driven test with verbosity enabled, we see that sub-tests are shown hierachically.

$ go test
=== RUN   TestAdd
=== RUN   TestAdd/Add(1,_1)
=== RUN   TestAdd/Add(-5,_3)
=== RUN   TestAdd/Add(9999,_-9999)
--- PASS: TestAdd (0.00s)
    --- PASS: TestAdd/Add(1,_1) (0.00s)
    --- PASS: TestAdd/Add(-5,_3) (0.00s)
    --- PASS: TestAdd/Add(9999,_-9999) (0.00s)
PASS
ok  	playground/add	0.006s

Server testing

Unit test is important but it’s never enough because it runs on some parts of the system separately. Integration testing or end-to-end testing will be really useful for us to build the confidence that the whole system works well together. To write integration testing on HTTP server, we can use net/http/httptest package, it has some convenient utilities.

First, let create a simple API endpoint that calculates sum of two integers.

type AddRequest struct {
	X int
	Y int
}

type AddResponse struct {
	Result int
}

func serveAdd(w http.ResponseWriter, r *http.Request) {
	switch r.Method {
	case http.MethodPost:
		var req AddRequest
		dec := json.NewDecoder(r.Body)
		dec.DisallowUnknownFields()
		err := dec.Decode(&req)
		if err != nil {
			log.Printf("Got body %+v", r.Body)
			http.Error(w, "Malformed request body", http.StatusBadRequest)
			return
		}
		log.Printf("Received add request %+v", req)

		resp := AddResponse{req.X + req.Y}
		w.Header().Set("Content-Type", "application/json")
		json.NewEncoder(w).Encode(resp)
	default:
		http.Error(w, "Method not allowed", http.StatusMethodNotAllowed)
	}
}

func main() {
	mux := http.NewServeMux()
	mux.HandleFunc("/add", serveAdd)
	log.Fatal(http.ListenAndServe(":8080", mux))
}

After starting the service, we can verify that the API endpoint is ready to serve requests.

$ curl 'localhost:8080/add' -H 'Content-Type: application/json' -d '{ "x": 431, "y": 87 }' | jq
{
  "Result": 518
}

How to write test for our server? We have to simulate the request and examine the response to see if it matched with our expection. There are two approaches which are setting up a test server or just writing test for hander s function only.

Set up a test server

httptest.NewServer returns a new testing Server which is an HTTP server listening on a system-chosen port on the local loopback interface. We can use this to set up a server that contains a single handler function.

func TestAddHandler(t *testing.T) {
	ts := httptest.NewServer(http.HandlerFunc(serveAdd))
	defer ts.Close()

	addReq := AddRequest{9, 32}
	var body bytes.Buffer
	json.NewEncoder(&body).Encode(addReq)
	resp, err := http.Post(ts.URL, "application/json", &body)
	if err != nil {
		t.Fatalf("Failed to call add request %v", err)
	}

	want := AddResponse{41}
	var got AddResponse
	json.NewDecoder(resp.Body).Decode(&got)
	if want != got {
		t.Errorf("Failed: got %v want %v", got, want)
	}
}

Write test for handler function

Because handler function needs request and response as parameters, we need to create both of them using httptest package, especially the response. httptest has ResponseRecorder type that capture response data for later inspection.

func TestAddHandler2(t *testing.T) {
	addReq := AddRequest{9, 32}
	var body bytes.Buffer
	json.NewEncoder(&body).Encode(addReq)
	req := httptest.NewRequest("POST", "/add", &body)
	resp := httptest.NewRecorder()

	serveAdd(resp, req)

	if resp.Code != http.StatusOK {
		t.Errorf("Got status %v, want %v", resp.Code, http.StatusOK)
	}
	want := 41
	var got AddResponse
	json.NewDecoder(resp.Body).Decode(&got)
	if got.Result != 41 {
		t.Errorf("Got result %v, want %v", got.Result, want)
	}
}

Both two examples cover single successful request for demonstration only, it’s recommendation that we should use table-driven testing method to verify all possible cases of the target endpoint.

Test in parallel

A useful feature for any testing libraries is the ability to execute list of tests in parallel. Using Go’s testing package, we can enable this feature with a simple setup. Usually, table-driven testing adds multiple tests at once, so using parallel feature will bring down the test running time a lot.

Let start with adding a simple function with a fake 2 seconds delay and along with its test.

func AddInt(x, y int) int {
	// Fake large processing time
	time.Sleep(2 * time.Second)
	return x + y
}

func TestAddInt(t *testing.T) {
	data := []struct {
		x, y, want int
	}{
		{0, 0, 0},
		{1, 0, 1},
		{56, 65, 121},
		{416, 2, 418},
		{-2, -88, -90},
		{-7, 34, 27},
		{-753, 753, 0},
	}

	for _, tc := range data {
		t.Run(fmt.Sprintf("x:%v, y:%v", tc.x, tc.y), func(t *testing.T) {
			got := AddInt(tc.x, tc.y)
			if got != tc.want {
				t.Errorf("AddInt failed, got %v, want %v", got, tc.want)
			}
		})
	}
}

If we run this test normally, the total test running time is sum of all sub-tests in the test because all the sub-tests are run sequentially.

$ go test -run TestAddInt
PASS
ok  	golang/playground/add	14.027s

Because sub-tests are independent with each other, using parallel testing won’t affect the test logic and reduces test running time significantly. Parallel testing can be enabled simply by adding t.Parallel() inside the sub-test.

t.Run(fmt.Sprintf("x:%v, y:%v", tc.x, tc.y), func(t *testing.T) {
	tc := tc // capture range variable
	t.Parallel()
	got := AddInt(tc.x, tc.y)
	if got != tc.want {
		t.Errorf("AddInt failed, got %v, want %v", got, tc.want)
	}
})

When using table-driven testing method with parallel testing is enabled, it’s necessary to create a new local variable to capture the current test data. If we don’t do that, all the sub-tests will run with the last value.

Let’s run the test again with verbosity flag on to see what’s going on with parallel testing enabled.

go test -run TestAddInt -v

That’s a huge optimization, test running time is reduced from ~14 seconds to ~2 seconds now. The total running time is roughly equal to the running time of a single sub-test because now all sub-tests are run in parallel. It does not mean that for all parallel tests, the total running time is roughly equal to single sub-test, it depends on number of test that Go can run in parallel. By default, the maximum number of test can be run in parallel is equals to GOMAXPROCS value which by default is the number of cores in the CPU. My machine has 8 cores while above test only has 7 sub-tests, that’s why I can have optimal running time which is equal to running single sub-test alone.

We can set the maximum number of test can be run in parallel by using flag parallel as in the bellow example. It took around 6 seconds to run because 7 tests were divided into 3 groups, each group tooks 2 seconds to run.

$ go test -run TestAddInt --parallel 3
PASS
ok  	golang/playground/add	6.029s

In the verbosity mode, we can see that a test have multiple states, from RUN to PAUSE then CONT and PASS. PAUSE and CONT are included in the output from release 1.10. A parallel test, at first, starts with RUN state, then it will be paused to wait for all non-parallel tests to complete (within a test function scope). It will continue execution with all other parallel tests later. Before release 1.6, running time for a parallel test is counted only for its execution, but now it also counts the time from the start of the test till t.Parallel() is called.

Race detection

Concurrency is an excelent approach when we want our softwares run faster to bring good UX to users, also we want to ultilize our server’s resources. Go is popular with its concurrency support by introducing goroutine feature. There are always to sides of anything and concurrency is not an exception. Concurrency brings many benefits but also leads to hours debugging to see the root cause of the problem because concurrency’s bugs are almost hard to see rightaway.

Let’s see the following trivial program

func increase_number(val, amount int) int {
	for i := 1; i <= amount; i++ {
	        // Let's use goroutine to make the operation go faster :D
		go func() {
			val++
		}()
	}
	return val
}

func main() {
	i := 1
	amount := 3000
	i = increase_number(i, amount)
	fmt.Printf("Final value of i is %v\n", i)
}

Everytime we run this trivial program, we will failed to see the expected result, and worst, each time, different output appear.

This situation is call data race, so what is data race in Go anyway? data race is a situation where at least two goroutine access a shared variable at the same time, and at least one goroutine tries to modify the variable.

Go is awesome when it supports data race detection simply by adding race flag when running or testing Go programs, when this flag is enabled, race detector will run against the current program.

$ go run -race main.go
$ go test -race main_test.go

Let’s run above example again with race flag enabled.

$ go run -race main.go 
==================
WARNING: DATA RACE
Read at 0x00c000136008 by goroutine 9:
  main.increase_number.func1()
      main.go:8 +0x38

Previous write at 0x00c000136008 by goroutine 7:
  main.increase_number.func1()
      main.go:8 +0x4e

Goroutine 9 (running) created at:
  main.increase_number()
      main.go:7 +0x87
  main.main()
      main.go:17 +0x44

Final value of i is 3000
Found 2 data race(s)

As you can see, Go excellently detects data race issues with the code example. The output is clear and precise, enough for us to debug and fix data race issues in our code. I recommend enable this race flag before submiting the PR for changes also for running test in CI to ensure that no possible data race issues while the coding is running on production.

Testing with interfaces

Coming from other languages, Go’s interface seems to be strange and easily to get wrong and often being used as in other languages.

Go interfaces generally belong in the package that uses values of the interface type, not the package that implements those values. The implementing package should return concrete (usually pointer or struct) types: that way, new methods can be added to implementations without requiring extensive refactoring.

— https://github.com/golang/go/wiki/CodeReviewComments#interfaces

So interfaces are instead of defined by implementing package, it should be defined by the package that uses it. This mean Go gives the power for the one who really needs it by letting them define their own interface that suits their needs.

Despite how excellent the way interface is used, there always a big question on testing that caused many developers a headache. How can we write test without interfaces from the package I use in my code? or sometimes it will be like, should you give me the interfaces for this package so I can write test for it?

Again, let me quote from Go guidelines

Do not define interfaces on the implementor side of an API “for mocking”; instead, design the API so that it can be tested using the public API of the real implementation.

— https://github.com/golang/go/wiki/CodeReviewComments#interfaces

New programmers come to Go and usually write code like this, it’s similar with what happens in other languages, especially Java. I used to write Java a lot too.

// add.go
package add

type Adder interface {
	AddInt(x, y int) int
}

type add struct{}

func NewAdder() Adder {
	return add{}
}

func (add) AddInt(x, y int) int {
	return x + y
}

// calculation.go
package calculation

type Service struct {
	addOperator add.Adder
}

func NewService(adder add.Adder) *Service {
	s := Service{adder}
	return &s
}

func (s Service) AddTwoInteger(x, y int) int {
	return s.addOperator.AddInt(x, y)
}

// main.go
package main

func main() {
	adder := add.NewAdder()
	calService := calculation.NewService(adder)
	sum := calService.AddTwoInteger(4, 8)
	fmt.Printf("%v + %v = %v\n", 4, 8, sum)
}

Leaving the main.go unchanged, we can refactor above code from implementing package defines interface to the package that uses it defines needed interfaces.

// add.go
package add

type Adder struct{}

func NewAdder() Adder {
	return Adder{}
}

func (Adder) AddInt(x, y int) int {
	return x + y
}

// calculation.go
package calculation

type Adder interface {
	AddInt(x, y int) int
}

type Service struct {
	addOperator Adder
}

func NewService(adder Adder) *Service {
	s := Service{adder}
	return &s
}

func (s Service) AddTwoInteger(x, y int) int {
	return s.addOperator.AddInt(x, y)
}

Alright, how can we write test for it?

If we want to mock the behaviour of a package, just simply define a mock struct which implements the interface that testing package needs, then use that mock for our testing without touching the real package.

// calculation_test.go
package calculation

import "testing"

type mockAdder struct{}

func (mockAdder) AddInt(x, y int) int {
	// We can fake some behaviour here
	return x + y
}

func TestAddTwoInteger(t *testing.T) {
	adder := mockAdder{}
	calService := calculation.NewService(adder)
	want := 5
	got := calService.AddTwoInteger(6, -1)
	if got != want {
		t.Errorf("Test failed, got %v, want %v", got, want)
	}
}

When talking about interface in Go, please remember following

Accept interface, return struct

Benchmark

Beside normal test, Go have built-in feature for running benchmarks to examine the performance of Go code. Benchmark function accept parameter of *testing.B instead of *testing.T as in normal test. Also, the target benchmark function must run the target code b.N times.

func fibonacci() func() int {
	x, y := 0, 1
	return func() (r int) {
		r = x
		x, y = y, x+y
		return
	}
}

func Fib(n int) []int {
	var r []int
	fib := fibonacci()
	for i := 0; i < n; i++ {
		r = append(r, fib())
	}
	fmt.Println(r)
	return r
}

func BenchmarkFibonacciClosure(b *testing.B) {
	for i := 0; i < b.N; i++ {
		Fib(100)
	}
}

Benchmark command is similar to command that is used to execute test, but with bench flag included.

$ go test -bench=.
goos: darwin
goarch: amd64
BenchmarkFibonacciClosure-8   	 1246626	       954 ns/op
PASS
ok  	benchmark-test	2.071s

Benchmark output means BenchmarkFibonacciClosure has been run 1246626 times at a speed of 954ns per iteration.

Benchmark also provides several options to run along, such as benchtime to set the benchmark running time or benchmem to print memory allocations.

$ go test -bench=. -benchtime=10s -benchmem=true
goos: darwin
goarch: amd64
BenchmarkFibonacciClosure-8   	12408841	       954 ns/op	    2088 B/op	      11 allocs/op
PASS
ok  	benchmark-test	12.828s

Test with different package

Most of the time, test file has the same package name as in the source file, this allow test file to avoid import package from source files for test cases, also, non-exported identifier can also be called in the tests because they are in the same package.

There are also some exceptions when test file does not share the same package name with source file, but being prepended with a suffix _test to the package name. With this package name, test file can only access exported identifier, but this allows us to write tests from user’s perspective, the one who uses our public packages.

Choosing package name same as in source file or have suffix _test is depends on whether we want to perform whitebox or blackbox testing.

There is a minor notice for test files that have suffix _test.

Test files that declare a package with the suffix _test will be compiled as a separate package, and then linked and run with the main test binary.

— https://tip.golang.org/cmd/go/#hdr-Test_packages
